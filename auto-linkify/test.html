Auto Linkify — 本地测试

1. 基础 URL 识别

用例 1.1 — HTTP
纯文本 http 链接应被识别为可点击链接
这是一段包含链接的文本 http://example.com/path?q=1 后面还有文字。
用例 1.2 — HTTPS
https 链接也应被识别
安全的链接 https://github.com/user/repo 在这里。
用例 1.3 — 多个 URL 在同一段文本
同一文本节点中的多个链接应都被转换
看看这两个链接：https://a.com/1 和 https://b.com/2 哪个好用？

2. URL 与中文混合

用例 2.1 — URL 后紧跟中文
URL 后面的中文不应被吞入链接
请访问 https://www.example.com/page了解一下。
用例 2.2 — URL 前后都有中文
中文环境中的链接应正确提取
这是链接https://www.baidu.com/哦，可以试试看。
用例 2.3 — 全角标点
全角逗号“，”不应被包含在 URL 中
参考链接：https://example.com/doc，还有更多资料。

3. Skip Zone（不应处理）

用例 3.1 — <a> 标签
已有链接中的文本不应被二次处理
https://already.link/
用例 3.2 — <code> 标签
代码中的 URL 不应被转换
请查看配置：const url = "https://example.com/api";
用例 3.3 — <pre> 标签
预格式化文本中的 URL 不应被转换
npm install --registry https://registry.npmjs.org
用例 3.4 — <textarea>
输入框内的 URL 不应被转换

4. Shadow DOM 支持

用例 4.1 — 自定义元素 Shadow DOM
Shadow DOM 内的文本节点应该被处理
用例 4.2 — 嵌套 Shadow DOM
嵌套 shadow root 内的链接也应被处理

5. 动态内容（MutationObserver）

用例 5.1 — 动态添加的 URL
JS 动态插入的文本中的 URL 应被处理
等待动态内容……
用例 5.2 — 动态添加到 Skip Zone 内的 URL
动态插入到 <code> 内的 URL 不应被处理
等待……

6. 边界情况

用例 6.1 — 不含 URL 的普通文本
无 URL 的文本应保持原样
这是一段完全没有任何链接的普通中文文本。
用例 6.2 — 纯数字/邮箱（不匹配的模式）
email 和 ftp 协议当前不应被转换
联系邮箱 user@example.com，或者 ftp://files.example.com（当前不处理这些）。
用例 6.3 — 嵌套 Skip Zone
<pre><code> 嵌套中的 URL 不应被处理
<a href="https://nested.example.com">link</a>
用例 6.4 — URL 在 <button> 内
按钮文本中的 URL 不应被处理
